# 项目质量核查报告

核查日期：2026-05-15

## 核查结论

当前项目已经从“手动账号录入工具”升级成了一个可继续演进的本地内容中台底座：账号监控池、半自动监控工作台、公开信息源采集、来源健康检查、每日简报都已经具备基本闭环。双账号定位、799 元 AI口播智能体、流量/信任/转化判断和合规边界都已经进入文档和核心逻辑。

但当前最明显的问题是：公开信息源采集结果仍然偏“泛 AI 资讯摘要”，还没有稳定变成“可直接拍的短视频选题”。部分条目与 AI先锋 / AI先锋者 的业务相关度偏低，标题和前 3 秒钩子模板化，日报里还存在“同一条内容既被推荐又被放进不建议做”的冲突。

### 2. AI先锋 / AI先锋者 在公开简报中的区分还不稳定

不达标。

表现：

- 日报中“今日更适合 AI先锋 的选题”和“今日更适合 AI先锋者 的选题”主要从 AI 消息表筛选，热点素材表没有充分参与账号适配分组。
- “两个都适合”的内容没有稳定同时进入两个账号的候选区。
- `analyzeFit()` 里把 agent、workflow、工具、课程等较容易判成“转化”，导致“信任”类内容偏少。
- AI先锋者 的信任/认知表达输出不足，很多内容只是“背后普通人入局AI要看懂这个信号”，还没有形成真正的行业判断或观点角度。

应改文件：

- `src/publicSourceCollector.js`

应改逻辑：

- `buildDailyBrief()` 里 AI先锋 / AI先锋者 分组应同时读取 `aiRows` 和 `hotRows`。
- “两个都适合”要同时进入两个账号的候选列表，但在文案里分别给出 AI先锋 版和 AI先锋者 版角度。
- `goal` 不应只按关键词硬判。建议拆成两个字段：`内容用途` 和 `承接方式`。
- AI先锋者 选题要优先输出“观点、判断、误区、机会、趋势”，而不是只输出通用标题模板。

  - 不适合承接原因

### 4. 每日简报有推荐和不推荐互相冲突的问题

不达标。

表现：

- `outputs/ai_daily_brief_2026-05-15.md` 中同一批 OpenAI News 条目既出现在 Top 5 和产品/工具变化里，又出现在“今日不建议做的热点及原因”里。
- 原因是 `riskNote()` 默认输出“发布时仍需核验时效和上下文”，而 `buildDailyBrief()` 的 `riskyRows` 用“核验”命中，导致几乎所有正常条目都被放进“不建议做”。

应改文件：

- `src/publicSourceCollector.js`

应改逻辑：

- `riskNote()` 默认的“需核验时效”不应等同“不建议做”。
- `riskyRows` 只应该收录真正风险项：
  - 传闻 / 爆料 / 未证实
  - 法律 / 监管 / 安全
  - 明确弱相关
  - 明确不建议做
  - 来源读取失败或摘要不足
- “需核验”应放在风险备注或发布前检查，不应进入“不建议做”区。

### 5. 标题和前 3 秒钩子可用性不足

不达标。

应改文件：

- `src/publicSourceCollector.js`

应改逻辑：

- `shortTitle()` 不要直接截断英文原题，应先提炼中文主题。
- `hookLine()` 应按内容用途生成不同钩子：
  - 流量：反常识 / 痛点 / 结果差异

  - 转化：痛点 / 场景 / 可落地方法
- `buildAngle()` 应避免重复标点，并输出更像短视频脚本入口的结构。
- 对英文来源应增加“中文化选题转换”步骤，而不是把英文标题直接塞进标题模板。

## 当前最需要优先修正的 3 个问题

### 优先级 1：修正公开信息源筛选和不推荐冲突

为什么优先：

这是当前影响最大的问题。只要主表继续混入低相关资讯，日报就会泛化；只要推荐和不推荐冲突，用户就无法信任简报。

应该改哪个文件：

- `src/publicSourceCollector.js`

应该改什么逻辑：

- 重写或增强 `isRelevant()`。
- 增加 `businessRelevanceScore`，围绕 AI口播、内容生产、智能体、自动化、工具应用、AI创业、AI变现、自媒体提效、知识付费、短视频运营、私信转化、直播承接打分。
- 主表只保留 A/B 级内容。
- 修正 `riskyRows`，不要把默认“需核验”当成“不建议做”。
- 把弱相关内容标为“不入主表”或进入低相关结果，而不是和有效内容混在一起。

### 优先级 2：把标题、前 3 秒钩子、切入角度改成真正可拍

为什么优先：

用户最终要的是内容中台，不是新闻摘要。当前日报虽然有标题和钩子字段，但可直接拍摄程度不够。

应该改哪个文件：

- `src/publicSourceCollector.js`

应该改什么逻辑：

- 优化 `shortTitle()`：先提炼中文主题，再生成标题，不直接截断英文。
- 优化 `hookLine()`：按流量 / 信任 / 转化生成不同钩子。
- 优化 `buildAngle()`：输出“标题、前3秒、账号适配、用途、799承接、下一步动作”。
- 对 AI先锋 输出更偏“普通人如何落地、如何提效、如何引出产品”。
- 对 AI先锋者 输出更偏“趋势判断、误区拆解、行业机会、为什么值得信任”。

### 优先级 3：增强 799 元 AI口播智能体承接判断

为什么优先：

AI先锋 的核心不是只追热点，而是把流量、信任和转化最终导向 799 元 AI口播智能体。当前字段已经有，但不足以指导成交链路。

应该改哪个文件：

- `src/publicSourceCollector.js`
- `src/analyzer.js`
- `docs/05-output-table-standards.md`

应该改什么逻辑：

- 新增或细化产品承接字段：
  - 是否适合承接 799 元 AI口播智能体
  - 适合承接方式
  - 产品连接句
  - 私信/直播/教程/案例中的哪一种更合适
  - 不适合承接原因
- AI先锋 的选题默认要判断是否能自然进入私信或直播。
- AI先锋者 的选题默认先沉淀信任，不强行卖产品。

- 对平台内容仍走监控池、人工补充、未来授权可见页面整理的路线。

需要后续继续注意的表达：

- “每天监控抖音等平台爆款风向”应始终写成“基于人工补充或授权可见页面整理的平台风向”。
- “采集平台内容”应避免单独出现，建议统一写成“公开信息源采集”或“授权页面可见信息整理”。
- “自动拆解爆款视频不少于 20 条”要明确样本来源来自公开来源、人工补充、用户授权可见页面或已有本地数据。

## 总体判断

项目方向正确，底层数据流已经成型，合规边界清楚，双账号框架也已经落到主要逻辑中。

下一步不应该开发新模块，也不应该进入插件或平台自动读取阶段。最应该先把公开信息源输出从“泛 AI 新闻摘要”打磨成“可拍、可筛、可承接、可复盘”的选题系统。

优先改完公开信息源的筛选、标题钩子和 799 承接判断后，这个工具才会真正从“能自动出表”进入“能自动辅助做内容决策”的阶段。
